# Документационный Styleguide

Этот документ задаёт единый стиль для текстов в `README`, `CONTRIBUTING`, шаблонах `Issue/PR` и страницах в `docs/`.

## 1. Tone of Voice

- Уверенный, доброжелательный, технический.
- Пишем просто и по делу: сначала польза, потом детали.
- Без маркетинговых клише и размытых формулировок.

## 2. Терминология

Используем единообразно:

- **MCP-сервер** (не «прокси», не «бот»)
- **инструмент** (`ask`, `research`, `reason`)
- **вложение** (`file_paths`)
- **cookie-строка** и **JWT-токен**
- **интеграционные тесты** / **юнит-тесты**

Правило: если термин уже закреплён в `README`, в других файлах не переименовывать.

## 3. Заголовки, абзацы, списки

- Иерархия без пропусков: `##` → `###` → `####`.
- Абзацы 2–4 строки.
- Для последовательности используем нумерованные списки.
- Для наборов фактов — маркеры.

## 4. Таблицы

Используем таблицы для:

- сравнений инструментов,
- матриц совместимости,
- переменных окружения,
- SLA/политик.

Таблица должна отвечать на один конкретный вопрос.

## 5. Код и команды

- Всегда fenced-блоки с указанием языка (`bash`, `json`, `python`).
- Команды должны быть копируемыми без редактирования.
- Секреты не публикуем; используем заглушки вида `"..."`.

## 6. Ссылки

- Внутренние ссылки — относительные (`./CONTRIBUTING.md`, `./docs/STYLEGUIDE.md`).
- Внешние ссылки только на проверенные источники.
- Не добавляем «битые» или несуществующие ссылки.

## 7. Callouts и акценты

Разрешённые типы:

- `NOTE` — важный контекст
- `TIP` — практический совет
- `IMPORTANT` — ограничение/инвариант
- `WARNING` — риск/последствия

Не использовать callout ради декоративности: только если реально помогает.

## 8. Markdown-возможности GitHub

Используем уместно:

- `details/summary` для вторичных пояснений,
- task list для чеклистов,
- Mermaid-диаграммы для архитектурных потоков,
- автоссылки на issue/PR/commit при наличии.

## 9. Контент-качество (чек перед merge)

- [ ] Есть чёткая структура и навигация.
- [ ] Формулировки короткие и однозначные.
- [ ] Нет пустых секций и шаблонного «шума».
- [ ] Команды и примеры проверяемы.
- [ ] Терминология совпадает с `README`.
- [ ] Markdown корректно рендерится на GitHub.
